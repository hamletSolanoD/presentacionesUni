Mocha

Framework de Pruebas para JavaScript

Flexible, potente y popular

Orígenes e Inicios

  • Lanzado en 2011 por TJ Holowaychuk
  • Proyecto Open Source bajo OpenJS Foundation
  • Licencia MIT
  • Ampliamente respaldado por la comunidad JavaScript

Instalación básica:

npm install -g mocha
Mencionar que TJ Holowaychuk es un prolífico desarrollador de JavaScript, conocido por múltiples proyectos populares.

Motivación de su creación

Antes de Mocha:

  • No había un estándar claro para pruebas en Node.js
  • Soporte limitado para pruebas asíncronas

Mocha buscaba:

  • Proporcionar un marco simple y flexible
  • Facilitar pruebas de código asíncrono (callbacks, promises)
  • Mejorar el flujo de trabajo de testeo

Extendió ideas de otros marcos (como Jasmine o QUnit) pero optimizado para Node.js

En la documentación se presenta como un "marco de pruebas JavaScript muy popular" que organiza y ejecuta casos de prueba

¿Qué es y qué problemas resuelve?

Framework de pruebas unitarias de JavaScript que permite:

  • Escribir y ejecutar tests de forma ordenada
  • Probar eficientemente funciones asíncronas
  • Generar reportes flexibles de resultados

Se definen bloques describe e it (BDD/TDD style) para agrupar y detallar pruebas

No impone librerías de aserción ni estilo concreto: puedes usar assert, Chai, Sinon u otra biblioteca de tu elección

Características clave:

  • Ejecución secuencial de tests
  • Manejo automático de errores no capturados
  • Asignación de errores al test correcto

¿Dónde y cómo se utiliza?

Entornos

  • Proyectos Node.js (backend)
  • Navegadores (con bundlers)
  • Aplicaciones web
  • APIs
  • Librerías JavaScript

Integración

  • Pipelines CI/CD (Travis, Jenkins)
  • NPM scripts
  • Ejecutar con: mocha
  • Directorio por defecto: test/

Instalación y uso básico:

  1. Instalar la dependencia mocha en el proyecto
  2. Definir directorio de tests (por defecto test/)
  3. Ejecutar mocha desde línea de comandos
  4. O agregar como script de npm, p.ej:
{
  "scripts": {
    "test": "mocha"
  }
}

Instalación y primer uso

Instalación típica:

npm init -y           # Inicializar proyecto Node
npm install --save-dev mocha

Ejecutar tests:

$ mocha

Output típico de un test exitoso:

$ mocha
.      # (se muestra un punto)
1 test complete (1ms)

Ejemplo práctico de testeo

Módulo a probar (sum.js):

function sum(a, b) {
  return a + b;
}
module.exports = sum;

Test (test/sum.test.js):

const sum = require('../sum');
const assert = require('assert');

describe('sum()', function() {
  it('debería sumar dos números', function() {
    assert.strictEqual(sum(2, 3), 5);
  });
});

Ejemplo práctico (salida)

Al ejecutar mocha:

  sum()
    ✓ debería sumar dos números

  1 passing (5ms)

✓ indica que la prueba pasó

Si la prueba falla, mostraría un error indicando la diferencia entre valor esperado y real

La salida está coloreada: verde para éxito, rojo para fallos

Integraciones y características adicionales

Agnóstico en librería de aserciones:

  • Chai: para expect/should
  • Sinon: para mocks/espías

Características avanzadas:

  • Hooks (before, after, etc.) para setup/teardown global o por caso
  • Configuración por archivo (.mocharc.js)
  • Reportes integrables (XUnit, JSON, cobertura de código)

Facilita estructurar tests complejos y se integra bien con el ecosistema JavaScript moderno (Node, Babel, TypeScript, etc.)

Recursos y enlaces útiles

  • Tutorial en español: DigitalOcean (Mocha y Assert)
  • Video Tutorial: "Pruebas con Mocha y Chai" - YouTube: JoanMedia

Resumen de Mocha

"Framework flexible y sencillo para desarrolladores JavaScript"
  • Ideal para desarrolladores JavaScript que necesiten un marco de pruebas flexible y sencillo
  • Facilita la implementación de TDD/BDD en Node.js y browser
  • Filosofía de libertad al desarrollador (assertions, estructura)
  • Amplio soporte de comunidad lo hace muy popular
  • Aprender Mocha prepara para trabajar con muchas herramientas de testing en JS
  • Principales ventajas: sencillez, manejo nativo de async y capacidad de reporting personalizado

Mockito

Framework de Mocking para Java

Simple, limpio y potente

Orígenes e Inicios

  • Creado por Szczepan Faber (ThoughtWorks) alrededor de 2008
  • Basado en proyectos de la Guardian (Londres)
  • Proyecto open source mantenido por la comunidad
  • El autor explicó su necesidad de un nuevo framework tras usar EasyMock
  • Actualmente mantenido por Szczepan, Brice Dutheil, Rafael Winterhalter y otros
  • Es gratuito, licencia MIT, y se distribuye a través de Maven Central o Gradle
  • Para empezar, simplemente se agrega la dependencia mockito-core en el proyecto
Mencionar que surgió como alternativa a EasyMock, buscando una API más limpia.

Motivación de su creación

Mockito surgió para ofrecer:

  • Una API más sencilla y legible que la de EasyMock (otro framework de mocks)
  • Simplificar el proceso de crear objetos simulados (mocks) en pruebas unitarias
  • Evitar configuraciones verbosas

Mockito permite:

  • Simular comportamientos de métodos en objetos dependientes de forma limpia
  • Verificar interacciones con métodos
"Un framework con una API simple y limpia que evita errores confusos al escribir tests"

Facilita el desarrollo de pruebas TDD/BDD en aplicaciones Java

¿Qué es y qué problemas resuelve?

Mockito es un framework de mocking para Java que permite:

  • Crear mocks (objetos simulados) en tests unitarios
  • Probar unidades de código que dependen de otras clases
  • Reemplazar dependencias (bases de datos, servicios externos, etc.) por dobles de prueba

Características principales:

  • Definir comportamiento de métodos simulados: when(...).thenReturn(...)
  • Verificar llamadas a métodos: verify(...)
  • Aislar la clase bajo prueba
  • Confirmar lógica interna sin requerir el entorno real
  • Mejorar la legibilidad de las pruebas comparado con soluciones anteriores

¿Dónde y cómo se utiliza?

Entornos

  • Pruebas unitarias de Java
  • Junto con JUnit (o TestNG)
  • Proyectos Java empresariales
  • Microservicios (incluyendo Spring Boot)
  • Probar servicios, controladores y componentes que dependen de otros

Integración

  • Dependencia org.mockito:mockito-core en el build (Maven/Gradle)
  • Anotación @Mock para declarar simulaciones
  • @InjectMocks para inyectarlas
  • Alternativa: Mockito.mock()
  • Ejecución con runner de JUnit normal; no requiere comando especial

Al correr las pruebas, Mockito se encarga internamente de generar los mocks y validar las interacciones

Instalación y configuración

Maven:

<dependency>
  <groupId>org.mockito</groupId>
  <artifactId>mockito-core</artifactId>
  <version>5.3.1</version>
  <scope>test</scope>
</dependency>

Gradle (Kotlin DSL):

testImplementation("org.mockito:mockito-core:5.3.1")

Activar con JUnit 5:

@ExtendWith(MockitoExtension.class)
public class MiTest { ... }

Ejemplo práctico

Clase a probar:

public class Calculadora {
  private ServicioExterno servicio;
  
  public Calculadora(ServicioExterno servicio) {
    this.servicio = servicio;
  }
  
  public int duplicarSuma(int a, int b) {
    int suma = servicio.sumar(a, b);
    return suma * 2;
  }
}

Interfaz dependiente:

public interface ServicioExterno {
  int sumar(int x, int y);
}

Ejemplo práctico (test)

Test con Mockito:

@ExtendWith(MockitoExtension.class)
public class CalculadoraTest {
  @Mock private ServicioExterno mockServicio;
  @InjectMocks private Calculadora calc;

  @Test
  public void testDuplicarSuma() {
    // Configurar el comportamiento del mock
    when(mockServicio.sumar(2, 3)).thenReturn(5);
    
    // Ejecutar el método a probar
    int resultado = calc.duplicarSuma(2, 3);
    
    // Verificar el resultado
    assertEquals(10, resultado);
    
    // Verificar la interacción con el mock
    verify(mockServicio).sumar(2, 3);
  }
}

Ejemplo práctico (salida)

Al ejecutar el test con JUnit:

CalculadoraTest > testDuplicarSuma() PASSED
Tests run: 1, Failures: 0, Errors: 0, Skipped: 0

Mockito intercepta la llamada a mockServicio.sumar(2,3) y devuelve 5

JUnit valida que assertEquals(10, resultado) pase

verify confirma que el método fue llamado con los argumentos correctos

Mockito actúa tras bambalinas, sin necesidad de servicio real

Características adicionales

Tipos de objetos de prueba:

  • Mocks, Spies, Stubs, Captive mocks

Funcionalidades avanzadas:

  • Simular comportamientos de métodos: thenReturn, thenThrow
  • Verificar llamadas: cantidad, orden, argumentos
  • Capturar argumentos con ArgumentCaptor
  • Modos de llamada básica: strict, lenient
  • Matchers para validaciones flexibles: any(), eq()

Documentación y extensiones:

  • Documentación oficial incluye buena guía de best practices
  • Al ser un proyecto muy usado, existen muchas extensiones y guías
  • Integración con Spring: anotación @MockBean
  • JUnit 4: MockitoJUnitRunner

Recursos y enlaces útiles

  • Documentación oficial: site.mockito.org y javadoc.io
  • Tutorial en español: PRUEBAS UNITARIAS en JAVA (JUNIT 5)
  • Repositorio GitHub: mockito/mockito

Resumen de Mockito

"Facilita las pruebas unitarias en Java al permitir simular dependencias de forma simple"
  • API limpia mejora la legibilidad de tests complejos
  • Al dominar Mockito, los desarrolladores pueden crear tests más robustos y centrados en la lógica de negocio
  • Permite aislar entornos externos
  • Estándar en proyectos Java modernos (Spring, Microservicios)
  • Complementa marcos de test como JUnit
  • En conjunto con Mocha (para JS), cubre necesidades fundamentales de testing en aplicaciones web full-stack
  • Ofrece confiabilidad y calidad en el software

Comparación

Mocha

  • Framework de pruebas completo
  • Enfocado en ejecutar tests
  • Necesita librerías adicionales para assertions
  • Excelente para pruebas asíncronas
  • Ecosistema JavaScript

Mockito

  • Framework de mocking
  • Enfocado en simular dependencias
  • Complementa JUnit
  • Excelente para aislar componentes
  • Ecosistema Java

Ambos son esenciales para garantizar código de calidad en sus respectivos ecosistemas
